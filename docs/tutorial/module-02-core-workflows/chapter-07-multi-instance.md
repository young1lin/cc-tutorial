## 为什么要使用多个实例？

### 单实例的限制

使用单个 Claude Code 实例时：

```
┌─────────────────────────────────────────────┐
│           单实例工作流                        │
├─────────────────────────────────────────────┤
│                                             │
│  任务 A → 等待 → 任务 B → 等待 → 任务 C    │
│                                             │
│  问题：                                     │
│  - 必须等待一个任务完成才能开始下一个        │
│  - 上下文可能混淆                            │
│  - 没有交叉验证                              │
│                                             │
└─────────────────────────────────────────────┘
```

### 多实例的优势

```
┌─────────────────────────────────────────────┐
│           多实例工作流                        │
├─────────────────────────────────────────────┤
│                                             │
│  实例 1: 任务 A ━━━━━━━━━━━  ✓             │
│  实例 2:     任务 B ━━━━━━━━  ✓             │
│  实例 3:         任务 C ━━━━  ✓             │
│                                             │
│  优势：                                     │
│  - 并行执行，节省时间                        │
│  - 实例间可以交叉验证                        │
│  - 专注不同的任务                            │
│  - 不会混淆上下文                            │
│                                             │
└─────────────────────────────────────────────┘
```

## 关于多 Opus 窗口的真相

### 常见误解

❌ **误解**: "不能同时运行多个 Opus 窗口"

### 真相

✅ **真相**: 技术上完全可以同时运行多个 Opus 窗口

**限制主要是配额和定价，而非技术限制**

### 什么限制了多窗口？

| 限制类型 | 说明 | 解决方案 |
|----------|------|----------|
| **API 配额** | 每个账户有并发请求数限制 | 升级 API 计划 |
| **成本** | 多窗口消耗更多 tokens | 合理使用，按需调整 |
| **上下文隔离** | 每个实例独立运行 | 这是特性，不是 bug |

### 专家如何使用多窗口

```
Zvi Mowshowitz 的设置：
- 5-10 个并行窗口
- 所有窗口并行运行
- 始终使用 Opus 4.5 with Thinking
- 每个窗口专注不同任务

Boris Power 的设置：
- 使用 git worktrees
- 每个 worktree 一个 Claude 实例
- 同时开发多个独立功能
```

## 模式 1: 一个编写，一个审查

### 工作原理

```
┌─────────────────────────────────────────────────────────┐
│           "一个编写，一个审查"模式                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  实例 A: 编写代码                                       │
│  ────────────────────                                  │
│  1. Claude A 编写功能实现                               │
│  2. 将代码保存到临时文件                                │
│                                                         │
│  实例 B: 审查代码                                       │
│  ────────────────────                                  │
│  3. Claude B 读取代码                                   │
│  4. Claude B 审查并提供反馈                             │
│                                                         │
│  实例 A: 修改代码                                       │
│  ────────────────────                                  │
│  5. Claude A 根据反馈修改                               │
│  6. 重复直到满意                                        │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 具体步骤

#### 步骤 1: 启动两个终端

```
# 终端 1 - 编写实例
cd /path/to/project
claude

# 终端 2 - 审查实例
cd /path/to/project
claude
```

#### 步骤 2: 在实例 A 中编写代码

```
> [终端 1 - 实例 A]

请实现一个用户头像上传功能，包括：
- 图片验证（大小、格式）
- 压缩图片
- 上传到云存储
- 返回图片 URL

请编写实现代码。
```

**Claude A** 编写代码...

```typescript
// src/api/avatar.ts

export async function uploadAvatar(userId: string, file: File): Promise<string> {
  // Validate file
  if (!file.type.startsWith('image/')) {
    throw new Error('File must be an image');
  }

  if (file.size > 5 * 1024 * 1024) {
    throw new Error('File size must be less than 5MB');
  }

  // Compress image
  const compressed = await compressImage(file);

  // Upload to cloud storage
  const url = await uploadToStorage(compressed);

  return url;
}
```

#### 步骤 3: 在实例 B 中审查代码

```
> [终端 2 - 实例 B]

请审查 src/api/avatar.ts 中的代码。

检查：
1. 错误处理是否完整
2. 是否有安全问题
3. 是否有性能问题
4. 代码风格是否一致
5. 是否缺少重要的功能
```

**Claude B** 的审查反馈：

```
## 代码审查反馈

### 问题
1. **安全性**: 没有验证文件内容（可能伪装成图片的恶意文件）
2. **性能**: 没有处理并发上传的情况
3. **错误处理**: compressImage 可能失败，但未处理
4. **功能缺失**: 没有删除旧头像的功能

### 建议
1. 添加文件内容验证（magic number 检查）
2. 添加用户级别的上传锁
3. 添加重试机制
4. 实现旧头像清理

### 代码风格
✓ 符合项目规范
✓ 类型定义清晰
✓ 命名有意义
```

#### 步骤 4: 在实例 A 中修改代码

```
> [终端 1 - 实例 A]

根据审查反馈修改代码：
1. 添加文件内容验证
2. 添加上传锁
3. 添加错误处理和重试
4. 实现旧头像清理
```

**Claude A** 修改代码...

#### 步骤 5: 重复审查（可选）

```
> [终端 2 - 实例 B]

请重新审查修改后的代码。
```

### 这个模式的优势

1. **交叉验证**: 减少错误和疏漏
2. **分离关注**: 编写和审查分开
3. **更高质量**: 第二个视角发现问题
4. **持续改进**: 迭代到满意为止

## 模式 2: 使用 Git Worktrees

### 什么是 Git Worktrees？

**Git worktrees** 允许你同时检出同一个仓库的多个分支到不同的目录。

```
project/                    (主分支: main)
├── src/
├── tests/
└── ...

project-feature-a/          (分支: feature-a)
├── src/
├── tests/
└── ...

project-feature-b/          (分支: feature-b)
├── src/
├── tests/
└── ...
```

### 为什么使用 Worktrees？

```
不使用 worktrees:
- 需要频繁切换分支
- 每次切换都要重新编译
- 不能同时开发多个功能
- 容易产生未提交的更改冲突

使用 worktrees:
- 每个功能独立目录
- 可以同时开发多个功能
- 不会互相干扰
- 每个 worktree 可以运行独立的 Claude 实例
```

### 创建 Worktrees

```bash
# 1. 在主项目中创建 worktree
cd project
git worktree add ../project-feature-a feature-a
git worktree add ../project-feature-b feature-b

# 2. 现在你有三个目录
project/           (main 分支)
project-feature-a/ (feature-a 分支)
project-feature-b/ (feature-b 分支)

# 3. 列出所有 worktrees
git worktree list

# 4. 完成后删除 worktree
git worktree remove ../project-feature-a
```

### 在每个 Worktree 中运行 Claude

```bash
# 终端 1 - 功能 A
cd ../project-feature-a
claude

# 终端 2 - 功能 B
cd ../project-feature-b
claude

# 终端 3 - 主分支（修复 bug）
cd project
claude
```

### 完整示例：并行开发三个功能

```
┌─────────────────────────────────────────────────────────┐
│         使用 Worktrees 并行开发                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Worktree 1: feature/user-search                       │
│  ─────────────────────────────────                     │
│  Claude 任务:                                           │
│  - 实现用户搜索 API                                     │
│  - 创建搜索组件                                         │
│  - 编写测试                                             │
│                                                         │
│  Worktree 2: feature/avatar-upload                     │
│  ─────────────────────────────────                     │
│  Claude 任务:                                           │
│  - 实现头像上传功能                                     │
│  - 集成云存储                                           │
│  - 添加图片压缩                                         │
│                                                         │
│  Worktree 3: main (hotfix/bug-fix)                     │
│  ─────────────────────────────────                     │
│  Claude 任务:                                           │
│  - 修复登录 bug                                        │
│  - 添加日志                                            │
│  - 部署热修复                                          │
│                                                         │
│  结果：三个功能同时开发，互不干扰！                      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 模式 3: 实例间通信

### 让 Claude 实例互相交流

你可以让不同的 Claude 实例通过文件进行通信：

```
┌─────────────────────────────────────────────────────────┐
│              实例间通信模式                              │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  实例 A:                                               │
│  ─────────────                                         │
│  1. 制定计划                                           │
│  2. 将计划写入 plan.md                                 │
│                                                         │
│  实例 B:                                               │
│  ─────────────                                         │
│  3. 读取 plan.md                                       │
│  4. 根据计划编写代码                                   │
│  5. 将进度写入 progress.md                              │
│                                                         │
│  实例 A:                                               │
│  ─────────────                                         │
│  6. 读取 progress.md                                   │
│  7. 监督进度，提供反馈                                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 实际示例

#### 实例 A: 规划者

```bash
# 终端 1
cd project
claude
```

```
> 请为添加用户个人资料功能制定详细计划。
将计划写入 docs/plan.md。

[Claude A 创建计划]
✓ 计划已写入 docs/plan.md
```

#### 实例 B: 执行者

```bash
# 终端 2
cd project
claude
```

```
> 请阅读 docs/plan.md 中的计划。
然后开始实现：从第 1 步开始，每完成一步更新 docs/progress.md。

[读取 plan.md]
开始实现第 1 步...
[完成]
更新 progress.md...
```

#### 实例 A: 监督者

```
> 请阅读 docs/progress.md 查看进度。
如果有问题，提供反馈。

[读取 progress.md]
进度很好，但第 2 步需要调整...

将反馈写入 docs/feedback.md
```

#### 实例 B: 根据反馈调整

```
> 请阅读 docs/feedback.md 中的反馈。
调整实现并继续。

[读取 feedback.md]
调整实现...
继续...
```

## 高级技巧

### 技巧 1: 使用不同的模型

```
实例 1: 使用 Opus 4.5（规划）
实例 2: 使用 Sonnet 4.5（编码）
实例 3: 使用 Opus 4.5（审查）
```

**优势**: 节省成本，同时保持质量

### 技巧 2: 专业化分工

```
实例 1: 专注前端
实例 2: 专注后端
实例 3: 专注测试
实例 4: 专注文档
```

**优势**: 每个实例成为"专家"

### 技巧 3: 使用 iTerm2/Windows Terminal 的标签页

```bash
# macOS (iTerm2)
# 为每个 worktree 创建一个标签页
# 设置通知，当 Claude 需要关注时提醒

# Windows (Windows Terminal)
# 使用标签页组织不同实例
# 设置自动标题显示当前工作目录
```

### 技巧 4: 并行运行相似任务

```
场景: 需要将 10 个文件从框架 A 迁移到框架 B

实例 1: 迁移文件 1-3
实例 2: 迁移文件 4-6
实例 3: 迁移文件 7-10

使用脚本来分配任务，让每个 Claude 实例处理一部分。
```

## 常见问题

### Q1: 会不会太贵？

**A**: 取决于使用方式：
- ✅ 合理使用：成本可控，价值远超成本
- ✅ 使用不同模型：Opus 用于规划，Sonnet 用于编码
- ✅ 并行独立任务：节省的时间远多于成本

### Q2: 会不会太难管理？

**A**: 一些技巧：
- ✅ 使用 git worktrees 组织代码
- ✅ 使用终端标签页组织会话
- ✅ 给每个实例明确的任务
- ✅ 定期检查进度

### Q3: 实例之间会冲突吗？

**A**: 使用 worktrees 不会：
- ✅ 每个实例有独立的工作目录
- ✅ Git 操作独立进行
- ✅ 合并时像正常分支一样处理

### Q4: 需要多少个实例？

**A**: 根据任务复杂度：
- 🟢 简单任务：1-2 个实例
- 🟡 中等任务：2-3 个实例
- 🔴 复杂任务：3-5 个实例（像 Zvi 那样）

## 总结

### 关键要点

1. **多实例完全可行**
   - 技术上无限制
   - 配额和成本是主要考虑

2. **三种主要模式**
   - "一个编写，一个审查"
   - 使用 git worktrees 并行开发
   - 实例间通信协作

3. **专家都在用**
   - Zvi: 5-10 个并行窗口
   - Boris: 使用 worktrees

4. **组织很重要**
   - 使用 worktrees
   - 使用终端标签页
   - 给每个实例明确任务
