## 什么是 Plan Mode？

### 定义

**Plan Mode** 是 Claude Code 中最重要的功能之一。它让 Claude 在**执行任何修改操作之前**，先制定一个详细的实现计划。

### 核心思想

```
┌─────────────────────────────────────────────────────────┐
│              正常模式 vs Plan Mode 对比                   │
├─────────────────────────────────────────────────────────┤
│  正常模式:                                               │
│  你 → Claude → 直接修改代码 ⚠️                          │
│                                                         │
│  Plan Mode:                                             │
│  你 → Claude → 制定计划 → 你审查 → 执行修改 ✅          │
└─────────────────────────────────────────────────────────┘
```

### 为什么 Plan Mode 如此重要？

> **"投入精力把验证机制做扎实。这是回报率最高的投资。"**
> —— Boris Power (Claude Code 之父)

Plan Mode 的价值：

1. **避免盲目修改**: Claude 不会"冲动"地修改代码
2. **策略正确性**: 先确认方向是否正确
3. **节省时间**: 避免后期大量返工
4. **学习机会**: 可以从计划中学习 Claude 的思路
5. **控制感**: 你始终保持对项目的控制

## 如何使用 Plan Mode

### 激活方式

**方法 1: 快捷键（推荐）**

```
按 Shift+Tab 一次 → 进入 Plan Mode
按 Shift+Tab 两次 → 退出 Plan Mode
```

**方法 2: 命令**

```
> /mode plan
已切换到 Plan Mode

> /mode auto
已切换到自动执行模式
```

**方法 3: 配置文件**

```json
// .claude/settings.json
{
  "permissions": {
    "defaultMode": "plan"  // 默认使用 Plan Mode
  }
}
```

## Plan Mode 工作流程

### 完整流程示例

```
┌─────────────────────────────────────────────────────────┐
│                    Plan Mode 工作流程                     │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. [你] 激活 Plan Mode (Shift+Tab)                      │
│       ↓                                                 │
│  2. [你] 描述需求                                        │
│       ↓                                                 │
│  3. [Claude] 分析代码库                                  │
│       ↓                                                 │
│  4. [Claude] 制定详细计划                                │
│       ↓                                                 │
│  5. [你] 审查计划 ← 关键步骤！                           │
│       ↓                                                 │
│  6a. [你] 批准 → [Claude] 执行                           │
│       ↓                                                 │
│  7. [Claude] 完成实现                                    │
│                                                         │
│  6b. [你] 要求修改 → [Claude] 调整计划 → 回到步骤 5      │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 实际示例

#### 步骤 1: 激活 Plan Mode

```
> [按 Shift+Tab]

✓ 已进入 Plan Mode
Claude 将只制定计划，不执行任何修改操作
```

#### 步骤 2: 描述需求

```

让我分析一下项目结构和相关文件...

[读取文件: src/components/]

基于分析，我制定以下实现计划：
```

```

### 第 1 步：创建个人资料页面组件
- 文件: src/pages/Profile.tsx
- 内容:
  - 用户信息展示区域

- 文件: src/api/profile.ts
- 端点:
  - GET /api/user/profile - 获取个人资料
  - PUT /api/user/profile - 更新个人资料
  - PUT /api/user/password - 修改密码

### 第 3 步：添加路由
- 文件: src/App.tsx
- 添加路由: /profile

### 第 4 步：创建密码修改对话框
- 文件: src/components/PasswordChangeDialog.tsx

### 第 5 步：集成到导航
- 文件: src/components/Navbar.tsx
- 添加"个人资料"链接

### 第 6 步：添加状态管理
- 使用 React Context 或 Redux
- 管理用户资料状态

是否需要调整此计划？
```

#### 步骤 5: 审查和修改计划

```
> 计划看起来不错，但我想用不同的方式处理头像上传

[调整计划...]

好的，我将修改头像上传方案：

## 何时使用 Plan Mode

### ✅ 应该使用的场景

#### 1. 多文件修改

```
需求: 重构认证系统，涉及 5+ 个文件

使用 Plan Mode: ✓
- Claude 可以规划整体架构
- 确保所有文件协调一致
- 避免遗漏某个文件
```

#### 2. 不熟悉的代码库

```
场景: 第一次接触新项目

使用 Plan Mode: ✓
- Claude 先探索代码库
- 理解现有架构
- 制定符合现有模式的计划
```

#### 3. 复杂功能实现

```
需求: 添加实时聊天功能（WebSocket、消息存储、UI）

使用 Plan Mode: ✓
- 需要多步实现
- 有多个技术方案
- 需要考虑边界情况
```

#### 4. 大型重构

```
需求: 将单体组件拆分为多个小组件

使用 Plan Mode: ✓
- 需要规划组件层次
- 确定状态管理方案
- 避免破坏现有功能
```

#### 5. 端到端任务

```
需求: 从数据库到 UI 完整实现新功能

使用 Plan Mode: ✓
- 涉及多个层次（数据库、API、UI）
- 需要协调前后端
- 需要考虑测试
```

## 何时不使用 Plan Mode

### ❌ 不应该使用的场景

#### 1. 快速的单词修正

```
> 将 "usre" 改为 "user"

使用 Plan Mode: ✗ 太慢了
直接修复: ✓ 更快
```

#### 2. 简单的代码格式化

```
> 格式化这个文件

使用 Plan Mode: ✗ 不必要
直接格式化: ✓
```

#### 3. 已非常熟悉的改动

```
> 添加一个 console.log 到这个函数

使用 Plan Mode: ✗ 过度规划
直接添加: ✓
```

#### 4. 需要低延迟的迭代

```
场景: 快速实验不同方案

使用 Plan Mode: ✗ 每次都规划太慢
直接尝试: ✓ 快速反馈
```

## 阅读和理解计划

### 计划的结构

Claude 生成的计划通常包含：

```
## 计划标题

### 概述
- 总体目标
- 涉及的主要组件

### 详细步骤

#### 第 X 步：[步骤名称]
- **文件**: [文件路径]
- **内容**: [具体内容]
- **原因**: [为什么这样做]

### 风险和注意事项
- [可能的问题]
- [需要特别小心的地方]

### 测试计划
- [如何验证]
```

### 理解关键信息

**关注点**:

1. **文件清单**
   ```
   将修改哪些文件？
   是否有遗漏？
   ```

2. **执行顺序**
   ```
   步骤顺序是否合理？
   是否有依赖问题？
   ```

3. **技术方案**
   ```
   使用的方法是否合适？
   是否符合项目现有模式？
   ```

4. **潜在风险**
   ```
   可能破坏什么？
   如何避免？
   ```

## 修改和优化计划

### 常见调整场景

#### 1. 添加缺失的步骤

```
> 计划中缺少错误处理部分

[调整计划...]

好的，我将添加：

### 第 X 步：添加错误处理
- 创建 src/utils/errorHandler.ts
- 实现统一的错误处理
- 添加用户友好的错误提示
```

#### 2. 改变技术方案

```
> 我不想用 Redux，想用 React Context

[调整计划...]

好的，我将修改为：

### 第 6 步：状态管理
- 使用 React Context API
- 创建 src/context/UserContext.tsx
- 避免引入 Redux 依赖
```

#### 3. 调整实现顺序

```
> 我觉得应该先写测试，再实现功能

[调整计划...]

好的，采用 TDD 方法：

### 第 1 步：编写测试
- 创建测试文件
- 定义期望行为

### 第 2 步：实现功能
- 按照测试要求实现
```

#### 4. 简化或扩展

```
> 这个功能太复杂了，我们先做基础版本

[调整计划...]

好的，简化为 MVP：

### 第 1 步：基础版本
- 只显示个人信息
- 简单的编辑功能

### 后续迭代（暂不实现）:
- 高级编辑功能
- 头像裁剪
- ...
```

## Plan Mode 最佳实践

### 1. 始终以 Plan Mode 开始复杂任务

```
复杂任务 → 先 Plan Mode → 审查 → 执行
```

### 2. 提供足够的上下文

```
❌ "添加一个功能"
✅ "添加用户个人资料页面，参考现有的设置页面结构"
```

### 3. 主动审查计划

```
不要只是快速浏览 → 认真检查每一步
- 文件是否正确？
- 方法是否合适？
- 是否有遗漏？
```

### 4. 不要害怕要求修改

```
计划不是绝对的 → 你可以调整
- 添加缺失的步骤
- 改变技术方案
- 调整实现顺序
```

### 5. 使用计划作为文档

```
好的计划可以作为：
- 实现指南
- 代码审查依据
- 团队沟通材料
```

1. 进入深度思考模式

```

Claude 会：
1. 启动子代理分析数据库
2. 收集详细信息
3. 基于分析结果制定计划
```

```

> [Plan Mode] 规划前端实现

# 终端 2
> [Plan Mode] 规划后端 API

两个实例可以并行规划不同的部分
```

## 常见问题

### Q1: Plan Mode 会增加多少时间？

**A**: 对于简单任务，可能增加 1-2 分钟。但对于复杂任务，**实际上节省时间**：
- 避免错误方向
- 减少返工
- 一次性完成

### Q2: 可以让 Claude 自动执行计划吗？

**A**: 可以。在 VS Code 扩展中，可以"一键批准"整个计划。但建议第一次手动审查。

### Q3: 如果计划有问题怎么办？

**A**: 直接告诉 Claude 问题所在，它会调整计划。这是 Plan Mode 的核心优势！

### Q4: Plan Mode 支持所有模型吗？

**A**: 是的，但更强大的模型（如 Opus）会制定更好的计划。

#### 不使用 Plan Mode

```
你: 添加用户认证功能

[创建文件]
[修改现有代码]
[... 30 分钟后 ...]

你: [测试发现]
- 缺少密码验证
- 没有 token 过期处理

```

2. 设计认证流程
3. 实现 JWT token 管理
4. 创建登录/注册页面
5. 添加密码验证
6. 实现 token 刷新机制
7. 添加错误处理
8. 编写测试

你: [审查计划]
计划看起来很全面，但我想用不同的 token 存储方式

Claude: [调整计划]
好的，将使用 localStorage + 内存缓存...

你: 批准，开始实现

Claude: [按计划执行]
[所有步骤按计划完成]

你: [测试]
✓ 所有功能正常
✓ 符合预期

结果: 一次性完成，质量高
```

## 总结

### 关键要点

1. **Plan Mode 是最重要的功能**
   - `Shift+Tab` 激活
   - 让 Claude 先思考后行动

2. **知道何时使用**
   - ✅ 多文件、复杂任务、不熟悉代码库
   - ❌ 快速修复、简单格式化

3. **认真审查计划**
   - 检查文件、顺序、方案
   - 不要害怕要求修改

4. **结合其他功能**
   - Thinking Mode、子代理、多实例

5. **投资回报率高**
   - 前期时间投入
   - 后期节省大量返工时间
